## What to ADD to Our Harness

### 1. Default-FAIL Contract (from `cwc-long-running-agents`)

**What it is**: A structural done-criteria that defaults to FAIL. Agents must **prove** completion with test results, screenshots, or verifiable evidence — not just claim "done."

**Why it matters**: Our current contract gates (G1-G4) define *when* to check, but not *what counts as proven*. The Default-FAIL contract is the missing half: the evidence standard.

**How to add**:
- Each gate already has checks. Add an **evidence requirement** column:
  - G1: "Requirements clear" → evidence: requirements.md written, no TBD items
  - G2: "Plan actionable" → evidence: plan.md has file paths + code snippets
  - G3: "Tests pass" → evidence: `pytest` / `pnpm test` exit code 0 in terminal output
  - G4: "Evaluator PASS" → evidence: evaluator posted PASS comment with diff summary
- Default state = FAIL. Agent must flip to PASS with evidence. No PASS without proof.

**Changes to team-roles**:
- `advisor.md`: Add evidence requirement to each gate checklist
- `worker.md`: Add "provide proof with every completion claim" rule
- `orchestrator.md`: Add "no PASS without evidence" enforcement

### 2. Fresh-Context Evaluator (from `cwc-long-running-agents`)

**What it is**: A separate agent with **no Write/Edit tools** that grades work from a fresh context window that never saw the build. Eliminates "I built this so it must be good" bias.

**Why it matters**: Our current Advisor has Write/Edit tools and shares context with the worker. The Anthropic finding is explicit: agents bias toward praising their own work. A read-only evaluator is structurally immune.

**How to add**:
- New role file: `.claude/team-roles/evaluator.md`
- Tool restriction: **read-only** (Read, Grep, Glob, Bash for running tests — no Write/Edit)
- Triggered by Orchestrator at G4 (post-implement evaluation)
- Reads: plan.md + generated code + test output. Produces: PASS/FAIL with checklist.
- This is **different from Advisor** (Opus, consultative) — Evaluator is read-only verification.

### 3. Initializer Agent Pattern (from "Effective Harnesses" Nov 2025)

**What it is**: The very first context window uses a **specialized prompt** that asks the model to set up the environment: `init.sh`, `progress file`, feature list with 200+ failing items, and an initial git commit.

**Why it matters**: Our design starts with Research → Plan but skips the **scaffold setup** step. The initializer creates infrastructure that all future sessions depend on.

**How to add**:
- New Phase 0.5 in the design doc: "Initialize" (before Research)
- Initializer agent creates:
  - `init.sh` — project-specific setup script (deps, env, build)
  - `features.json` — granular feature list, all marked `status: failing` by default
  - `PROGRESS.md` — rolling log of what each session accomplished
  - Initial git commit showing scaffold files
- The initializer does NOT plan features — it only sets up the team and scaffold.
- This maps to our `orchestrator.md` — add a "first session" section.

### 4. Context Window Anxiety Mitigation (from pattern catalog)

**What it is**: At 70%+ context usage, inject a reminder: "You have significant headroom remaining. Continue working thoroughly — do not rush or wrap up early."

**Why it matters**: Anthropic's own research shows models "exhibit context anxiety" — wrapping up work prematurely as they approach perceived limits. Our design mentions 70% threshold for context reset, but doesn't address the **behavioral** problem.

**How to add**:
- `orchestrator.md`: Add a turn hook that checks context % and injects a headroom reminder
- The reminder goes in the system prompt, not as a user message
- This is a **prompt engineering** fix, not a code change — fits naturally in team-roles

### 5. Agent-Maintained Handoff (from `cwc-long-running-agents`)

**What it is**: `PROGRESS.md`, `CLAUDE.md`, and a **stop hook** that automatically saves progress and uncommitted changes before context reset or session end.

**Why it matters**: Our design has `handoff.md` as an artifact, but doesn't specify **automatic** handoff. The Anthropic pattern uses a Claude Code stop hook to guarantee state persistence.

**How to add**:
- Add a Claude Code `Stop` hook that:
  1. Writes current progress to `PROGRESS.md`
 2. Stashes or commits uncommitted changes
 3. Updates `features.json` with completed items
- This ensures no work is lost even on unexpected session termination

## What to ADJUST in Our Harness

### 6. Tool Minimization (from Vercel d0)

**What they found**: Vercel's d0 agent had 15 specialized tools. Accuracy: 83%. They cut to 3 general tools. Accuracy: **95%**.

**What it means for us**: Our `worker.md` lists no tool restrictions. Every worker gets the full tool suite. More tools = more decision points = more wrong turns.

**Adjustment**:
- `worker.md`: Restrict to **Bash, Read, Write, Grep, Glob** (5 tools). No web search, no MCP, no browser.
- `researcher.md`: Gets **Read, Grep, Glob, WebSearch** (4 tools). No Write — research only.
- `evaluator.md`: Gets **Read, Grep, Glob, Bash** (4 tools). No Write/Edit — read-only verification.
- Advisor gets full tools (needs flexibility for judgment calls).

### 7. Chachamaru's 5-Verb Structure (from `claude-code-harness`)

**What it is**: setup → plan → work → review → release. Each "verb" is a skill file with clear entry/exit criteria. Go-native guardrail engine enforces rules at sub-10ms.

**What's worth borrowing**: Not the Go engine (too heavy), but the **verb-skill structure**. Our current team-roles are role descriptions, not workflow skills.

**Adjustment**:
- Add 5 skill files alongside team-roles:
  - `.claude/commands/harness-setup.md`
  - `.claude/commands/harness-plan.md`
  - `.claude/commands/harness-work.md`
  - `.claude/commands/harness-review.md`
  - `.claude/commands/harness-release.md`
- Each maps to a harness phase and can be invoked individually or chained
- Orchestrator calls these as sub-tasks within the `/goal` loop

### 8. Managed Agents Interface Concept (from Anthropic Apr 2026)

**What it is**: Anthropic's hosted agent service is built around **interfaces that outlast any particular harness implementation**. The key insight: harnesses encode assumptions that go stale as models improve.

**What it means for us**: Our contract gates (G1-G4) are good, but they're **tightly coupled to today's model capabilities**. As models improve, gate thresholds and retry budgets become wrong.

**Adjustment**:
- Add a **harness config** section to the design doc:
  - Gate retry budgets as tunable parameters (not hardcoded)
  - Model assignments as config (not baked into role files)
  - Context threshold as env var (not hardcoded 70%)
- This way, when Opus 5 ships and doesn't need 3 retries at G4, we just change the config.

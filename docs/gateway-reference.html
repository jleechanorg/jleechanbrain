Gateway (Local Machine) — Full Reference

Two gateways run as LaunchAgents:

| Gateway | Label | Port | State dir | Config |

|---|---|---|---|---|

| Production | ai.smartclaw.gateway | 18789 | ~/.smartclaw_prod/ | ~/.smartclaw_prod/hermes.json |

| Staging | ai.smartclaw.staging | 18810 | ~/.smartclaw/ | ~/.smartclaw/hermes.json |

~/.smartclaw/ = staging (the repo checkout). ~/.smartclaw_prod/ = production (separate dir, symlinks shared resources).

  • - Deploy: bash scripts/deploy.sh — validates staging canary, syncs config → prod, restarts prod, validates prod canary
  • - Status: launchctl print gui/$UID/ai.smartclaw.gateway | launchctl print gui/$UID/ai.smartclaw.staging
  • - Logs: ~/.smartclaw/logs/gateway.log (staging), ~/.smartclaw_prod/logs/gateway.log (prod)
  • - Liveness (required): curl -fsS -m 8 http://127.0.0.1:18789/health (prod) or :18810 (staging)
  • - hermes gateway health vs HTTP /health: Health uses WebSocket RPC (CLI default 10s). Under event-loop pressure (Slack socket, mem0, long runs), RPC can hit gateway timeout while curl /health still returns 200. For manual checks use hermes gateway health --timeout 30000. doctor.sh passes HERMES_DOCTOR_GATEWAY_RPC_TIMEOUT_MS (default 30000) to gateway status and gateway health.
  • - Liveness ≠ Functional (CRITICAL): HTTP /health returns {"ok":true,"status":"live"} even when the agent cannot authenticate. A gateway is only truly functional if (1) liveness passes AND (2) auth-profiles.json exists at $HERMES_STATE_DIR/agents/main/agent/auth-profiles.json. Missing auth-profiles → every Slack message silently fails with "No API key found for provider anthropic" while the health check looks green. Always run staging-canary.sh (check 8) or doctor.sh to verify functional state — never rely on /health alone after a deploy or plist migration.
  • - Hung gateway: launchctl stop gui/$UID/<label>, kill stray listener, then launchctl start gui/$UID/<label>
  • - Session lock silent failure (distinct from hung gateway): HTTP 200 but all Slack messages silently dropped. Symptom: gateway.err.log shows session file locked (timeout 10000ms). Cause: dead process held .lock files in ~/.smartclaw/agents/main/sessions/. Fix (handles both plain-PID and JSON {"pid":N} lock formats): find ~/.smartclaw/agents/main/sessions/ -name "*.lock" | while read f; do raw=$(cat "$f" 2>/dev/null); pid=$(echo "$raw" | python3 -c "import sys,json; print(json.load(sys.stdin)['pid'])" 2>/dev/null || echo "$raw" | tr -d '[:space:]'); [[ "$pid" =~ ^[0-9]+$ ]] && ! kill -0 "$pid" 2>/dev/null && rm -f "$f" && echo "removed: $f"; done — then restart gateway. HTTP 200 does NOT prove messages are processing — check err.log for session file locked after any restart.
  • - Lane backlog ("Slack feels dead" while /health is 200) — distinct from session locks and from raw WS churn counts: gateway.err.log shows [diagnostic] lane wait exceeded with large waitedMs and queueAhead ≫ 0. The main lane is saturated; inbound Slack work queues behind long embedded runs or stuck sessions. Do not assume a quiet gateway means a healthy user experience. Mitigations: (1) keep timeoutSeconds × maxConcurrent within the WS/event-loop discipline in this file; (2) clear stale locks, then restart the gateway to drain the queue; (3) confirm staging-canary.sh and a real Slack test message after restart. If Error ensuring model exists: … 404 repeats with mem0 failures, treat as model/plugin catalog drift — run hermes doctor against the correct HERMES_STATE_DIR and check gateway/plugin versions (not just curl /health).
  • - Staging LaunchAgent missing: If ~/Library/LaunchAgents/ai.smartclaw.staging.plist exists but launchctl print gui/$UID/ai.smartclaw.staging fails, staging is not loaded — curl :18810/health fails even though prod works. Fix: launchctl bootstrap gui/$UID ~/Library/LaunchAgents/ai.smartclaw.staging.plist then launchctl start gui/$UID/ai.smartclaw.staging (or re-run scripts/install-launchagents.sh). A missing staging daemon is not prod Slack failure, but operators often confuse the two when testing.
  • - CLI vs service config: The prod gateway uses HERMES_STATE_DIR=~/.smartclaw_prod and HERMES_CONFIG_PATH=~/.smartclaw_prod/hermes.json. Staging uses the staging plist/state dir. Any validation harness (doctor.sh, monitor-agent.sh, ad-hoc pytest around configs) must target HERMES_STATE_DIR / HERMES_CONFIG_PATH, never assume repo-root ~/.smartclaw/hermes.json.
  • - Deploy Stage 4 orphan kill: scripts/deploy.sh removes stray hermes-gateway PIDs before restarting prod. Exclusions are not just the lsof LISTEN pid — any PID in the ancestor chain of processes tied to ports 18810 (staging) or 18789 (prod) is protected so worker processes are not killed while the parent holds the socket.
  • - monitor-agent.sh: exit 0 doesn't mean full run — check for lock file skip (Another monitor-agent run is active)
  • - doctor.sh / monitor-agent.sh parity rule: Both scripts probe the same subsystems (gateway health, memory, session locks, heartbeat). When adding or changing a diagnostic probe in one, check the other handles the same case consistently and against the same live config path. Divergence causes one to FAIL while the other passes — creating false STATUS=PROBLEM. Run both after any probe change and compare results.
  • - hermes subcommand verification: Before writing any script that invokes an hermes subcommand, verify it exists: hermes <subcommand> --help 2>&1 | head -3. The command surface changes across versions. Do not hardcode memory vs mem0; inspect the installed CLI and prefer the configured plugins.slots.memory surface when scripting probes.
